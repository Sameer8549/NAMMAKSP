# Data Provenance

NAMMA KSP currently runs on a synthetically generated demonstration dataset. It
is not connected to live Karnataka State Police, CCTNS, court, telecom, banking,
or government identity systems.

## Included Demo Data

The backend loads CSV files from `data/` into SQLite during startup:

- `firs.csv`: synthetic FIR-like crime records with IDs, dates, districts,
  police stations, crime types, status, offender IDs, victim IDs, and location
  IDs.
- `offenders.csv`: synthetic offender profiles with age, gender, district,
  previous FIR count, and risk category.
- `victims.csv`: synthetic victim profiles linked to FIRs.
- `locations.csv`: synthetic district/station coordinates used for maps and
  hotspot views.
- `relationships.csv`: synthetic links between FIRs, offenders, victims, and
  locations for network analysis.
- `financial_transactions.csv`: small synthetic AML-style transaction examples
  linked to FIR IDs for prototype financial/cyber relationship analysis.
- `socio_economic_indicators.csv`: district-level synthetic/curated indicator
  fields used to demonstrate socio-economic correlation workflows.

These files are designed for evaluation, UI demos, analytics workflows, and
model behavior testing. They must not be treated as official police data.

## How The Demo Dataset Is Used

The app ingests the CSV files into normalized tables, then computes dashboard
KPIs, FIR search results, offender risk scores, network graphs, hotspot maps,
reports, and AI context from those local tables. AI responses are grounded in
retrieved database context and now include a `sources` field for citation.

## Real CCTNS/FIR Integration Path

A production integration would require formal data-governance approval and a
secure ingestion design:

1. Authorized API/file feed from CCTNS or the official FIR case-management
   system.
2. Field mapping from official FIR, accused, victim, station, case-status,
   evidence, and charge-sheet schemas into the NAMMA KSP analytical schema.
3. Data minimization rules so only required investigation fields are available
   to each role.
4. PII classification, encryption at rest, audit trails, retention periods, and
   deletion workflows approved by the responsible authority.
5. Operational validation with real investigators before any risk score,
   forecast, or early-warning alert affects decisions.
6. Production monitoring, incident response, and periodic bias/error review.

Until that integration exists, all screenshots, reports, AI answers, and
analytics generated by this repository should be understood as synthetic
prototype output.
